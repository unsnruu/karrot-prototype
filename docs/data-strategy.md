# 데이터 전략

## 문서 목적
이 문서는 현재 프로토타입 단계에서 DB와 Supabase를 어떤 역할로 이해해야 하는지 설명하기 위한 문서입니다.

이 문서는 아래 내용을 설명하기 위해 존재합니다.

- 왜 현재 단계에서 DB를 read 중심으로 다루는지
- 왜 완전한 CRUD를 필수 요건으로 보지 않는지
- 프로토타입 안정성과 데이터 변경 가능성 사이의 우선순위를 어떻게 두는지
- 화면 구현 시 어떤 수준의 상호작용이면 충분한지

이 문서는 아래 성격의 문서가 아닙니다.

- 최종 백엔드 아키텍처 확정 문서
- 운영 환경 설계 문서
- 세부 SQL 스키마 설명서

프로젝트의 제품 배경은 `docs/product-context.md`를 참고합니다.
구현 원칙은 `docs/implementation-principles.md`를 참고합니다.

## 현재 해석
이 프로젝트는 완전한 제품 구현이 아니라, 핵심 사용자 흐름과 실험 가설을 검증하기 위한 프로토타입입니다.

그래서 현재 단계에서 DB는 “실서비스와 동일한 동작을 모두 수용하는 백엔드”라기보다, “프로토타입 화면을 안정적으로 재현하기 위한 데이터 기반”에 더 가깝습니다.

즉, 현재 프로젝트에서 Supabase는 아래처럼 이해하는 것이 맞습니다.

- 현실감 있는 화면 맥락을 제공하기 위한 읽기 중심 데이터 소스
- 여러 사용자가 보더라도 기준 화면을 유지해야 하는 프로토타입용 기반
- 완전한 쓰기 경험보다 일관된 시나리오 재현을 우선하는 저장소
- 나중에 A/B 테스트에서 UI variation 을 공정하게 비교할 수 있게 해주는 기반

## 왜 read 중심인가
현재 단계에서 중요한 것은 사용자가 화면을 읽고 흐름을 이해할 수 있을 만큼의 맥락을 제공하는 것입니다.

특히 이 프로젝트는 최종적으로 어떤 UI가 `동네지도` 진입율을 더 높이는지 비교해야 하므로, variation 간 비교 맥락이 흔들리지 않는 것이 중요합니다.

반면 `Create`, `Update`, `Delete`를 실제로 넓게 열어두면 아래 문제가 생길 수 있습니다.

- 한 사용자의 상호작용이 다른 사용자가 보는 화면 상태를 바꾼다.
- 테스트 세션마다 기준 시나리오가 달라진다.
- 실험에서 비교해야 하는 화면이 불안정해진다.
- 시연용 프로토타입이 세션이 지날수록 의도와 다르게 훼손된다.

그래서 현재 우선순위는 “실제 쓰기 완성도”보다 “일관된 화면 상태 유지”에 있습니다.

## 현재 원칙
현재 단계에서는 아래 해석이 기본입니다.

- `Read`는 필요하다.
- `Create`, `Update`, `Delete`는 제한적이거나 생략될 수 있다.
- 상호작용은 보여주되, 반드시 영속 데이터 변경으로 이어질 필요는 없다.

즉, 프론트엔드와 백엔드가 모든 기능에서 완전한 양방향 CRUD로 연결될 필요는 없습니다.

## 구현 시 해석
위 원칙은 실제 구현에서 아래처럼 해석합니다.

- 실제 DB 조회를 통해 현실감 있는 리스트, 상세 화면, 채팅 맥락을 제공하는 것은 괜찮다.
- 좋아요, 예약, 댓글 작성, 수정, 삭제 같은 기능은 현재 단계에서 pseudo interaction 이어도 충분하다.
- 버튼을 눌렀을 때 즉각적인 UI 반응은 줄 수 있지만, 반드시 영속 저장까지 연결할 필요는 없다.
- 쓰기 기능이 필요해 보이는 경우에도 우선은 로컬 상태나 임시 상태로 시연 가능한지 먼저 본다.
- 기준 화면과 핵심 시나리오를 지키는 것이 실제 데이터 변경 가능성보다 더 중요하다.

## Supabase를 바라보는 현재 위치
현재의 Supabase는 아래 세 가지를 동시에 만족시키는 방향이 적절합니다.

- read 가능한 seed 데이터나 화면용 데이터를 제공한다.
- 프로토타입의 구조를 설명할 수 있을 만큼의 현실감을 준다.
- 나중에 필요하면 더 실제 백엔드에 가깝게 확장할 여지를 남긴다.

즉, 지금의 Supabase는 “모든 사용자 행동을 영속 저장하는 운영 시스템”이라기보다, “프로토타입을 읽기 가능하고 안정적으로 유지하는 기반”에 가깝습니다.

## 미래 확장 메모
나중에 프로젝트 목적이 바뀌어 실제 제품에 가까운 검증이 필요해지면, 쓰기 기능 범위를 점진적으로 넓힐 수 있습니다.

다만 현재 단계에서는 그 가능성을 열어두는 것만으로 충분하며, 지금 당장 완전한 CRUD를 전제로 구조를 무겁게 만들 필요는 없습니다.

## 한 문장 요약
현재 프로젝트의 DB는 완전한 운영 백엔드가 아니라, A/B 테스트 전 단계에서 비교 가능한 프로토타입 시나리오를 유지하기 위한 read 중심 데이터 기반으로 이해해야 합니다.
