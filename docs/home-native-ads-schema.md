# home_native_ads 최종 스키마

## 문서 목적
이 문서는 `home_native_ads` 테이블의 최종 컬럼 구성을 정리하기 위한 문서입니다.

이 문서는 아래 질문에 답하기 위해 존재합니다.

- 어떤 컬럼이 실제 테이블에 들어가는가
- 각 컬럼은 왜 필요한가
- 어떤 값 제한을 가져야 하는가

## 테이블 역할
`home_native_ads` 는 홈 피드 아이템 사이에 삽입되는 Native 광고 데이터를 저장하는 테이블입니다.

현재 기준의 역할은 아래와 같습니다.

- 홈 피드 인라인 광고의 텍스트와 이미지 정보를 담는다.
- 광고가 어떤 feature 로 연결되는지 정의한다.
- 홈 피드 안에서의 기본 노출 순서를 정의한다.

## 최종 컬럼

`id`
- 타입: `uuid`
- 역할: DB가 자동 생성하는 기본 키
- 메모:
- 입력 데이터 작성 시에는 따로 제공하지 않습니다.
- row 식별, 추후 클릭 이벤트 연결, 관리 편의를 위해 유지합니다.

`title`
- 타입: `text`
- 역할: 광고 카드의 메인 문구
- 메모:
- 홈 피드 문맥 안에 자연스럽게 섞일 수 있는 짧은 카피를 저장합니다.

`feature`
- 타입: `text`
- 역할: 광고 카드에 노출되는 feature 라벨
- 허용값:
- `동네지도`
- `동네 생활`
- `모임`
- `카페`

`image_url`
- 타입: `text`
- 역할: 광고 카드 썸네일 이미지 경로
- 메모:
- 현재는 비어 있을 수 있고, 이후 `supabase/images/home-native-ads/**` 경로를 채워 넣습니다.

`destination`
- 타입: `text`
- 역할: 광고 클릭 시 연결할 목적지 feature
- 허용값:
- `동네지도`
- `동네 생활`
- `모임`
- `카페`
- 메모:
- 현재는 `feature` 와 동일한 값을 넣는 것을 기본 규칙으로 사용합니다.
- 실제 URL path 는 앱 코드에서 이 값을 기준으로 매핑합니다.

`likes_count`
- 타입: `integer`
- 역할: 카드 하단 관심 수 표시
- 메모:
- 디자인상 숫자 표기가 필요하므로 유지합니다.

`placement_key`
- 타입: `text`
- 역할: 광고가 어떤 슬롯에 노출되는지 구분
- 현재 기본값:
- `home_feed_inline`
- 메모:
- 현재는 홈 피드 인라인 광고만 다루지만, 이후 placement 확장 가능성을 위해 유지합니다.

`sort_order`
- 타입: `integer`
- 역할: 같은 placement 안에서의 기본 정렬 순서
- 메모:
- 현재 홈 피드 as-is 기준인 `상품 3개마다 광고 1개` 삽입 시 어떤 광고를 먼저 쓸지 결정하는 기준이 됩니다.

`created_at`
- 타입: `timestamptz`
- 역할: 생성 시점 기록

`updated_at`
- 타입: `timestamptz`
- 역할: 수정 시점 기록

## 값 제한 규칙

`feature`
- 아래 네 값만 허용합니다.
- `동네지도`
- `동네 생활`
- `모임`
- `카페`

`destination`
- 아래 네 값만 허용합니다.
- `동네지도`
- `동네 생활`
- `모임`
- `카페`

`feature` 와 `destination`
- 현재는 동일한 값을 넣는 것을 기본 규칙으로 사용합니다.

`placement_key`
- 현재는 `home_feed_inline` 만 사용합니다.

## 구현 메모
- 실제 라우팅은 DB에 path 를 직접 저장하지 않고, 앱 코드에서 `destination` 값을 path 로 변환합니다.
- 현재 기준의 매핑은 아래와 같습니다.
- `동네지도` -> `/town-map`
- `동네 생활` -> `/community`
- `모임` -> `/community?tab=meetup`
- `카페` -> `/community?tab=cafe`

## 한 줄 요약
`home_native_ads` 는 홈 피드 인라인 광고의 카피, 이미지, feature 목적지, 노출 순서를 저장하는 테이블입니다.
