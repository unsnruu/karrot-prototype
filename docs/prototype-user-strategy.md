# 프로토타입 고정 유저 전략

## 문서 목적
이 문서는 현재 프로토타입 단계에서 `현재 사용 중인 사람`을 어떤 방식으로 해석하고 유지할지 설명하기 위한 문서입니다.

이 문서는 아래 내용을 설명하기 위해 존재합니다.

- 왜 고정된 프로토타입 유저가 필요한지
- 그 유저 정보를 DB에 둘지, 프론트에만 둘지에 대한 판단
- 현재 어떤 최소 구조를 채택했는지
- 앞으로 관련 기능을 구현할 때 어떤 기준을 따라야 하는지

이 문서는 아래 성격의 문서가 아닙니다.

- 최종 인증 시스템 설계 문서
- 운영 환경의 계정/권한 정책 문서
- 멀티유저 실시간 동기화 설계 문서

관련 배경은 아래 문서를 참고합니다.

- `docs/data-strategy.md`
- `docs/implementation-principles.md`
- `docs/product-context.md`

## 왜 필요한가
현재 프로토타입은 `read 중심`으로 안정적인 화면 맥락을 재현하는 것이 중요합니다.

하지만 아래 흐름에서는 단순한 seed 데이터만으로는 부족합니다.

- 채팅 목록과 채팅방
- 판매글 작성자 맥락
- 커뮤니티 글 작성자 맥락
- `나의 당근`처럼 현재 사용자 시점을 전제로 하는 화면

이런 화면은 결국 아래 질문에 답해야 합니다.

`지금 이 화면을 보고 행동하고 있는 사람은 누구인가?`

이 기준점이 없으면 채팅 상대 표기, 내가 올린 글, 내가 판매자인지 구매자인지 같은 기본 맥락이 흔들리게 됩니다.

## 현재 판단
현재 단계에서는 `고정된 프로토타입 유저 1명`을 두는 것이 맞습니다.

이 판단의 이유는 아래와 같습니다.

- 지금 프로젝트는 완전한 운영 백엔드보다 `일관된 시나리오 재현`이 더 중요합니다.
- 여러 테스트 세션에서 같은 사용자 시점을 유지해야 화면 해석이 쉬워집니다.
- 현재 UI는 이미 Supabase 기반 read 흐름을 일부 사용하고 있어서, 현재 유저 정보만 프론트 하드코딩으로 따로 들고 가면 데이터 소스가 갈라집니다.
- 반대로 현재 유저 정보를 DB에도 두면 상품, 글, 채팅 같은 맥락을 같은 기준으로 연결하기 쉬워집니다.

## DB에 둘지, 프론트에만 둘지
현재의 권장 방식은 아래 조합입니다.

- `유저 정보 자체는 DB에 둔다.`
- `지금 어느 유저를 현재 사용자로 볼지는 프론트의 상수로 둔다.`

즉, `public.users` 안에는 고정 프로토타입 유저가 실제 row로 존재하고, 프론트엔드는 그 유저의 `id`를 기준점으로 사용합니다.

이 방식을 선택한 이유는 아래와 같습니다.

- DB 기반 리스트/상세/채팅 데이터와 현재 사용자 정보가 같은 저장소를 참조하게 됩니다.
- 나중에 `내 글`, `내 채팅`, `내 판매글` 같은 읽기 로직을 만들 때 예외 처리가 줄어듭니다.
- 인증 시스템 없이도 현재 사용자 시점을 안정적으로 유지할 수 있습니다.

## 현재 최소 구현
현재는 아래 최소 구조를 채택합니다.

### 1. 고정 프로토타입 유저
- `public.users` 안에 프로토타입용 유저를 1명 둡니다.
- 현재 기준 유저는 `당근하트`입니다.
- 이 유저는 테스트와 시연에서 `현재 사용 중인 사람`으로 해석합니다.

### 2. 프론트 상수
- 프론트엔드에는 `현재 유저 id`를 반환하는 고정 상수를 둡니다.
- 이 상수는 화면이 현재 사용자 시점을 해석할 때 사용합니다.

### 3. 인증은 아직 도입하지 않음
- 현재 단계에서는 `auth.users` 기반 로그인 플로우를 필수로 보지 않습니다.
- 지금 필요한 것은 진짜 로그인보다 `시점이 고정된 사용자 맥락`입니다.

## 현재 DB 맥락
이번 판단 시점의 관련 상태는 아래와 같습니다.

- `public.users`에는 판매자/이웃 역할의 seed 유저들이 이미 존재합니다.
- `public.items`, `public.community_posts`, `public.businesses`는 read 중심 seed 데이터로 사용됩니다.
- `auth.users`는 현재 기준으로 비어 있을 수 있습니다.
- `chat_threads`는 현재 `buyer_name`만 갖고 있고 `buyer_id`는 없습니다.

즉, 현재 채팅 구조는 완전한 관계형 사용자 모델이라기보다, 화면 재현용 read 데이터에 더 가깝습니다.

## 채팅에 대한 현재 해석
현재 구조에서는 `chat_threads.buyer_name`이 채팅 상대를 표현합니다.

그래서 지금 단계에서는 아래처럼 해석합니다.

- 최소 변경이 필요하면 `buyer_name`을 고정 프로토타입 유저 이름에 맞춰 정리합니다.
- 아직 `buyer_id`를 추가하지 않아도 프로토타입은 충분히 성립할 수 있습니다.

다만 미래에는 아래 방향이 더 바람직합니다.

- `chat_threads.buyer_id`를 `public.users.id`와 연결한다.
- 이름 표시는 `buyer_id`에서 파생한다.

현재 문서는 어디까지나 `최소 변경` 기준을 설명합니다.

## 구현 가이드
앞으로 관련 기능을 구현할 때는 아래 규칙을 우선합니다.

- 현재 사용자 프로필 정보가 필요하면 프론트 임시 문자열을 새로 만들기보다 고정 프로토타입 유저를 참조합니다.
- `나의 당근`류 화면은 가능한 한 이 고정 유저를 기준으로 표현합니다.
- 내가 판매자인지 구매자인지 구분이 필요한 화면은 먼저 `현재 유저 id`와 existing row의 관계로 해석 가능한지 봅니다.
- 진짜 쓰기 기능이 없는 화면이라면 pseudo interaction 이어도 괜찮습니다.
- 인증 시스템이 필요하다는 이유만으로 지금 `auth.users` 중심 구조로 확장하지 않습니다.

## 비목표
현재 단계에서 아래는 아직 목표가 아닙니다.

- 회원가입/로그인 구현
- 여러 실제 사용자 계정 간 전환
- 사용자별 권한/세션/RLS를 중심으로 한 운영형 auth 구조
- 채팅/작성 기능의 완전한 영속 쓰기 흐름

## 한 문장 요약
현재 프로젝트에서는 `고정 프로토타입 유저를 public.users에 유지하고, 프론트에서는 그 유저 id를 현재 사용자 상수로 참조하는 방식`을 기본 전략으로 사용합니다.
