# 당근 프로토타입 업데이트 로그

## 문서 목적
이 문서는 지금까지 우리가 어떤 흐름으로 프로토타입 작업을 이어왔는지 한눈에 복기하기 위한 개인용 업데이트 로그입니다.

정확한 Git commit history를 대체하려는 문서는 아닙니다.
대신 아래를 빠르게 떠올리기 위해 존재합니다.

- 지금까지 작업의 큰 줄기가 무엇이었는지
- 왜 이 순서로 구현했는지
- 현재 어디에 집중하고 있는지
- 다음에 이어서 볼 때 어떤 맥락부터 다시 잡으면 되는지

## 사용 메모
이 문서는 공식 스펙 문서보다는 작업 흐름을 기록하는 운영 메모에 가깝습니다.

그래서 아래 원칙으로 업데이트합니다.

- 사실 관계보다 “작업 맥락”이 빠르게 보이도록 쓴다.
- 세부 구현보다 “왜 이 작업을 했는지”를 함께 남긴다.
- 새 작업이 시작되면 가장 위에 최신 로그를 추가한다.
- 하나의 로그는 길게 쓰기보다, `무엇을 했는지 / 왜 했는지 / 다음은 무엇인지` 정도만 남긴다.

## 현재까지의 작업 흐름 요약
지금까지의 큰 흐름은 아래처럼 정리할 수 있습니다.

1. 먼저 당근과 유사한 앱 구조를 빠르게 재현할 수 있는 프로토타입 뼈대를 만들었다.
2. 개별 기능을 깊게 파기보다, 하단 탭 전체 구조와 각 탭의 첫 진입 화면을 먼저 갖추는 방향으로 정리했다.
3. 이후 `홈`, `동네지도`, `상품 상세`, `채팅`처럼 실험 맥락상 중요한 흐름부터 depth를 조금씩 추가했다.
4. 데이터는 완전한 운영 백엔드보다 read 중심 프로토타입 구조로 유지하고, mock/local/Supabase를 섞더라도 UI 레이어는 비교적 안정적으로 유지하려고 했다.
5. 최종적으로는 `홈 -> 동네지도` 진입율을 높일 수 있는 UI 실험으로 수렴하도록 문서와 구현 방향을 맞추고 있다.

## 현재 상태 한 줄 요약
하단 탭과 주요 depth 2 흐름은 어느 정도 갖춰졌고, 지금은 `홈 안에서 동네지도로 자연스럽게 연결되는 진입 포인트`를 더 실험 가능한 형태로 다듬는 단계로 볼 수 있습니다.

## 2026-04-17
### 무엇을 했나
- `home -> town-map` 실험 라우팅을 다시 정리했다.
- 처음에는 `town-map`까지 `exp/a|b|c`로 이어지게 보강했지만, 다시 검토하면서 `home`만 variant 페이지로 유지하는 방향으로 바꿨다.
- 대신 `item detail`, `home services`, `sell flow`, 홈 탭 복귀처럼 `다시 홈으로 돌아오는 지점`들은 variant-aware 하게 맞췄다.

### 왜 이렇게 했나
- `community`, `town-map`, `chat`, `my-karrot`는 variant별 UI 차이가 없는데도 페이지를 모두 나누면 관리 대상이 불필요하게 늘어난다.
- 이 방식은 라우트 누락, 하위 페이지 누락, 링크 누락 같은 실수를 만들기 쉽고 실제로 `town-map`도 반쯤만 variant 라우트가 붙는 문제가 드러났다.
- 그래서 이번에는 `페이지 관리의 용이성`과 `오류 방지`를 기준으로, 실험이 실제로 적용되는 `home`만 variant로 두고 나머지는 공용 페이지로 유지하는 쪽이 더 낫다고 판단했다.

### 이번에 정리한 원칙
- `home`만 `/exp/{variant}/home` 으로 분기한다.
- 다른 탭은 공용 경로를 유지한다.
- 단, 사용자가 어디를 돌아다녀도 다시 홈으로 복귀할 때는 자신의 variant 홈으로 돌아간다.
- 이를 위해 `returnTo`와 홈 복귀 링크를 점검해서 상세 페이지와 연관 흐름에서도 variant가 끊기지 않게 했다.

### 다음에 이어서 보면 좋은 것
- 홈 실험 surface 자체가 더 늘어날 때도 같은 원칙을 유지할지
- `returnTo` 기반 복귀가 필요한 depth가 더 있는지
- 실험 종료 시 `exp/home` 정리 범위를 얼마나 단순하게 가져갈 수 있는지

## 2026-04-15
### 무엇을 하고 있나
- `docs` 안에 지금까지의 작업 흐름을 한눈에 볼 수 있는 업데이트 로그 문서를 추가했다.
- 현재 코드베이스 기준으로 보면 하단 탭 구조, 주요 화면 라우트, 여러 feature별 screen 분리가 이미 진행된 상태다.
- 최근 작업 축은 `홈` 피드 안에 들어가는 문맥형 진입 요소 쪽으로 이동해 있는 것으로 보인다.

### 왜 이 작업이 필요한가
- 문서가 많아질수록 전략 문서와 실제 작업 흐름 사이를 연결해 주는 가벼운 회고 로그가 필요했다.
- 나중에 다시 볼 때 “우리가 왜 이걸 먼저 만들었지?”를 빠르게 복구할 수 있어야 한다.
- 실제 커밋 로그는 세부 변경 중심이라, 제품 맥락까지 한 번에 읽히지는 않기 때문이다.

### 지금 보이는 프로젝트 상태
- 하단 탭 기준 주요 랜딩 화면은 대부분 구현돼 있다.
- `홈`, `커뮤니티`, `동네지도`, `채팅`, `나의 당근` 각각의 진입 페이지가 존재한다.
- `상품 상세`, `채팅 상세`, `동네지도 검색/상세`, `글 상세`, `판매 플로우` 같은 depth 2 이상 흐름도 일부 연결돼 있다.
- 구조적으로는 `app -> features/**/screens -> features/**/components` 방향이 비교적 잘 잡혀 있다.

### 현재 집중점 해석
- 이 프로젝트의 진짜 목적은 여전히 “당근 클론 완성”보다 `홈 -> 동네지도` 전환 실험에 가깝다.
- 그래서 최근 흐름은 홈 피드 안에서 쓸 수 있는 맥락형 진입 포인트를 만드는 쪽으로 읽힌다.
- 현재 워크트리에도 `home-native-ad` 관련 변경이 있어, 홈 피드 내 실험 surface를 구체화하는 단계로 해석할 수 있다.

### 다음에 이어서 보면 좋은 것
- 지금 로그 문서를 계속 누적형으로 쓸지
- 아니면 주차별 / 실험단위별 섹션으로 더 정리할지
- `홈 native ad`가 실제로 어떤 실험 가설을 대표하는지 별도 로그 항목으로 더 쪼갤지

## 이전 흐름 정리 메모
아래는 현재 코드와 기존 문서를 바탕으로 복기한 작업 흐름입니다.

### 1. 프로토타입 방향 정리
- 이 프로젝트를 “완성형 서비스”가 아니라 “읽기 중심 프로토타입”으로 해석하기 시작했다.
- 제품 목표를 `홈에서 동네지도로의 진입율 개선`으로 분명히 잡았다.
- 이를 위해 `product-context`, `roadmap`, `implementation-principles`, `architecture` 같은 기준 문서를 먼저 만들었다.

### 2. 하단 탭 구조 우선 구현
- 개별 기능을 깊게 구현하기 전에 앱 전체 IA를 먼저 보이게 하는 방향을 택했다.
- 각 하단 탭이 도달 가능하고, 첫 화면만 봐도 앱 구조가 설명되도록 만들었다.
- 이 단계에서 프로토타입의 “앱처럼 보이는 안정감”을 먼저 확보하려고 했다.

### 3. 중요 흐름만 선택적으로 깊게 확장
- 모든 탭을 똑같이 깊게 파지 않고, 실험 맥락상 중요한 흐름부터 확장했다.
- 홈 상품 상세, 채팅방, 동네지도 검색/상세, 커뮤니티 상세처럼 맥락을 설명해 주는 화면들을 추가했다.
- 판매 플로우처럼 데모 품질을 높여주는 보조 흐름도 일부 붙었다.

### 4. 데이터 소스와 구조 분리
- UI가 특정 mock 데이터 파일에 과하게 묶이지 않도록 조심했다.
- 완전한 CRUD보다 read 중심 흐름을 우선했고, 로컬 상태나 seed 데이터로도 충분한 구간은 가볍게 유지했다.
- 나중에 Supabase 비중이 커져도 screen 레이어를 다시 크게 뜯지 않도록 구조를 의식했다.

### 5. 실험 surface 구체화
- 프로젝트의 최종 목적이 `홈 -> 동네지도` 실험이기 때문에, 홈 안의 CTA surface를 어디에 둘지 점점 더 중요해졌다.
- 최근 방향은 홈 피드 안에 자연스럽게 섞이는 recommendation/entry module 계열로 모여 가는 중이다.
- 현재 워크트리도 그 흐름을 반영하고 있어, 이제는 실험안의 형태를 더 또렷하게 만드는 단계로 보인다.
